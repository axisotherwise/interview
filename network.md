# 네트워크
<br>

### 1. cors
서로 다른 오리진에서 데이터의 송수신을 허용하는 정책을 의미함. <br>
cors 에러는 sop가 다른 경우 정책에 위반되어 브라우저에서 발생시키는 에러임 <br>

1. 단순 요청
GET POST HEAD 메서드를 사용함. <br>

2. 사전 요청
Options 메서드를 사용함. <br>
실제 요청을 보내기 전 해당 리소스에 요청이 가능한지 확인하는 요청임. <br>

<br>

### 2. GET POST
- GET은 멱등성의 특징이 있으며 쿼리 스트링에 데이터를 담아 보내고 서버의 리소스를 조회할 경우 사용함. <br>
URL에 데이터가 노출되므로 보안적으로 중요한 데이터는 포함되면 안 됨. <br>

- POST는 바디에 데이터를 담아 보내며 서버의 리소스를 새로 생성하거나 업데이트할 경우 사용함. <br>

<br>

### 3. git
git branch 전략 중 하나인 git flow 전략을 사용함. <br>
크게 5개의 브랜치를 생성하여 관리함. <br>

1. 메인 브랜치 <br>
main: 현재 배포되어 있는 프로덕션 브랜치 <br>
develop: 다음 버전 출시를 위한 브랜치 <br>

2. 보조 브랜치(사용 후 삭제) <br>
feature: 기능 구현을 위한 브랜치 <br>
release: 테스트를 위한 브랜치 <br>
hotfix: 버그 수정을 위한 브랜치 <br>

<br>

### 4. XSS CSRF
브라우저에 스크립트를 실행할 수 있는 코드를 삽입하여 공격자가 의도한 스크립트를 실행하게 하는 공격임. <br>

- XSS는 사용자에서 실행됨.
- CSRF는 서버에서 실행됨.

<br>

### 5. WEB WAS
WEB 서버를 이용하여 리버스 프록시를 구성해 WAS 서버의 정보를 숨겨 보안성을 높일 수 있으며 로드밸런싱 기능이 가능함. <br> 

1. WEB
HTTP 요청으로부터 정적 리소스를 제공함. <br>
요청 인자 값에 상관없이 리소스가 변경되지 않음. <br>

2. WAS
HTTP 요청으로부터 정적 리소스를 제공함. <br>
대부분의 WAS는 WEB 서버의 기능을 내장하고 있음. <br>
요청 인자 값에 따라 리소스가 변경될 수 있음. <br>

<br>

### 6. 쿠키 세션
HTTP 프로토콜의 한계점을 보완하기 위해 등장함. <br>

1. 쿠키
브라우저 접속 시 생성되는 Key/Value 형태로 구성된 임시 파일임. <br>
브라우저에 저장되므로 임의로 수정/삭제가 가능하며 제3자에 의해 탈취되기도 쉬움. <br>

2. 세션
일정 시간 동안 같은 사용자로부터 들어오는 일련의 요청을 하나의 상태로 보고 해당 상태를 유지시키는 기술임. <br>
서버에 저장되어 쿠키에 비해 안전하지만 서버에서 다시 세션을 참조해야 하므로 성능은 쿠키에 비해 느림. <br>

<br>

### 7. 토큰
안전하게 데이터를 송수신하기 위해 JSON 객체 형태로 구성된 Claim 기반의 웹 토큰임. <br>
토큰은 서명되어 있으므로 신뢰가 가능하며 필요한 정보를 자체적으로 지니고 있는 자가 수용적 특징이 있음. <br>

1. 헤더
토큰의 타입과 서명 알고리즘의 정보임. <br>

2. 페이로드
사용자의 Entity(데이터)에 대한 설명임. <br>

3. 시그니처
헤더와 페이로드를 인코딩하며 토큰에 대한 유효성을 검증할 수 있음. <br>

<br>

### 8. OAUTH
사용자들이 사용 서비스에 비밀번호를 제공하지 않고 기존에 다른 웹 사이트에 저장돼있는 자신의 정보에 대한 접근 권한을 부여할 수 있도록 사용되는 수단임.

<br>

### 9. 세션 인증 토큰 인증
1. 세션 인증
클라이언트 요청 시 클라이언트의 상태를 저장하므로 StateFul 한 구조임. <br>

2. 토큰 인증
클라이언트 요청 시 클라이언트의 상태를 저장하지 않으므로 StateLess 한 구조임. <br>

- 단일 도메인이라면 세션 인증이 적합함. <br>
- 세션에 사용되는 쿠키는 단일 도메인 및 서브 도메인에서만 작동하도록 설계되어 있어 여러 도메인의 경우 관리하기 어려움.

<br>

### 10. Connection Timeout Read Timeout
1. Connection Timeout
서버 자체에 클라이언트가 어떠한 사유로 접근에 실패했을 경우를 의미함. <br>

2. Read Timeout
서버에 접근은 성공했지만 서버가 로직을 처리하는 시간이 너무 길어 발생함. <br>

<br>

### 11. REST
자원을 이름으로 구분해 해당 자원의 상태를 주고받는 아키텍처 스타일임. <br>
자원과 행위 그리고 표현으로 구성되어 있음. <br>
웹의 기술과 HTTP 프로토콜을 사용함. <br>

1. 자원은 HTTP URI로 구분함.
2. 행위는 HTTP Method로 구분함.
3. 표현은 데이터를 송수신하는 데이터의 형식을 의미함.

<br>

### 12. REST API
REST의 특징으로 API를 구성한 걸 의미함. <br>
각 요청이 어떤 동작이나 정보를 의미하는 건지 요청의 모습 자체로 추론이 가능해야 함. <br>

- URL에서 접근하는 리소스에 대해 명확하게 표현해야 함. <br>
- 표현하는 리소스에 동사가 들어가면 안 됨. <br>
- 자원은 URL에서 확인할 수 있고 행위는 메서드로 결과는 상태 코드로 확인함. <br>

<br>

### 13. HTTP 프로토콜
서버/클라이언트의 구조를 따르며 데이터 송수신을 위한 규칙임. <br>

1. 무상태성
클라이언트의 요청에 대한 상태를 저장하지 않음. <br>

2. 비연결성
요청에 맞는 응답을 보내면 연결을 끊음. <br>

<br>

### 14. HTTP HTTPS
1. HTTP 
TCP와 직접 통신하며 데이터를 평문으로 송수신하므로 제3자에 의해 조회될 수 있음. <br>

2. HTTPS
암호화된 데이터로 송수신하며 HTTP는 SSL과 SSL은 TCP와 통신 함.

<br>

### 15. TCP UDP
1. TCP
데이터 송수신을 위해 IP를 이용하는 연결형 프로토콜임. <br>
통신 간에 신뢰성을 보장함. <br>
3-way-handshake를 통해 연결하고 4-way-handshake를 통해 연결을 종료함. <br>

- 혼잡제어 가능

2. UDP
비연결형 프로토콜임. <br>
통신 간에 신뢰성을 보장하지 않음. <br>
IP를 그대로 사용해 단순 CheckSum 말고는 데이터의 훼손을 감지할 수 없음. <br>
중간에 데이터가 유실되어도 이를 다시 요청할 수 있는 방법이 없음. <br>

- 혼잡제어 불가능

<br>

### 16. URI URL
1. URI
네트워크 상의 자원을 나타내는 고유한 통합 자원 식별자임. <br>
정보 리소스를 고유하게 식별하고 위치를 지정함. <br>

2. URL
네트워크 상의 자원에 위치를 나타냄. <br>
리소스가 어디에 있고 어떻게 접근할 수 있는지를 의미함. <br>

<br>

### 17. 대칭키 비대칭키
1. 대칭키
암/복호화에 사용되는 키가 동일(대칭)함을 의미함. <br>
개인키로 데이터를 암호화하고 복호화 역시 암호화에 사용된 개인키를 이용해야함. <br>

2. 비대칭키
암/복호화에 사용되는 키가 다름(비대칭)을 의미함. <br>

<br>

### 18. 3-way-handshake
TCP/IP 프로토콜을 이용해 통신을 하는 응용 프로그램이 데이터를 송수신하기 전 먼저 정확한 전송을 보장하기 위해 상대방 컴퓨터와 사전에 세션을 수립하는 과정을 의미함. <br>

1. 클라이언트는 서버에게 접속을 요청하는 SYN 플래그를 보냄. <br>
2. 서버는 클라이언트에게 ACK와 SYN 플래그를 보냄. <br>
3. 클라이언트는 다시 서버에게 ACK 플래그를 보내고 연결이 이루어짐. <br>

<br>

### 19. 4-way-handshake
TCP/IP 프로토콜을 종료하기 위해 클라이언트와 서버가 연결된 세션을 종료하는 과정을 의미함. <br>

1. 클라이언트가 서버로 연결을 종료하겠다는 FIN 플래그를 보냄. <br>
2. 서버는 클라이언트에게 ACK 플래그를 보내고 통신 종료까지 기다림. <br>
3. 서버는 통신이 끝나면 FIN 플래그를 보냄. <br>
4. 클라이언트는 확인했다는 의미로 서버에게 ACK 플래그를 보냄 <br>

- Time Wait 이란 서버가 FIN 플래그를 보내는 과정에서 FIN 보다 데이터가 늦게 도착하는 경우 데이터 손실이 발생하는 걸 방지하기 위한 과정임. <br>

<br>

### 20. 프로세스 스레드
1. 프로세스
실행되는 프로그램 자체와 프로그램이 실행되는 주변 환경을 의미함. <br>
저장돼있는 프로그램이 메모리에 올라가 인스턴스화됨을 의미함. <br>

2. 스레드
프로세스 내부에 프로세스의 자원을 공유하거나 공유하지 않고 실행되는 작업의 단위를 의미함. <br>

<br>

### 21. 크롬 탭
크롬은 탭마다 PID를 가지고 있으며 개별 프로세스임. <br>
탭마다 렌더링 정보나 기타 데이터를 따로 가지고 있음. <br>
개별 프로세스라 메모리를 많이 사용하지만 하나의 탭에서 오류가 생겨도 다른 탭의 영향을 주지 않음. <br>
