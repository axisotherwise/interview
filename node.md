# 노드
<br>

### 노드
V8 엔진으로 빌드 된 자바스크립트 실행기입니다. <br>
싱글 스레드 논 블로킹 I/O 모델이며 이벤트 기반으로 작동합니다. <br>

* 노드는 서버 외의 용도로도 사용될 수 있습니다. (실행기)

<br>

### 노드 사용 이유
I/O 요청이 많을 경우 사용합니다. <br>
자바스크립트 언어로 서버를 구성할 수 있으며 라우터와 미들웨어 등 서버 구성에 필요한 대부분의 기능을 자체적으로 가지고 있는 프레임워크를 통해 짦은 시간 안에 효율적인 구성이 가능합니다. <br>

<br>

### 노드 작동 원리
노드의 내부는 V8 엔진과 libuv로 이루어져 있습니다. <br>

* V8 엔진은 콜 스택과 메모리 힙으로 구성되어 있습니다. <br>
CPU Bound(자바스크립트 코드) 작업을 동기적으로 실행합니다. <br>

<br>

* libuv는 C 언어로 구성된 라이브러리이며 이벤트 루프와 스레드 풀로 구성되어 있습니다. <br>
이벤트 루프를 통해 I/O Bound 작업을 비동기적으로 실행합니다. <br>

<br>

### 노드 장단점
* 장점 <br>
CPU 연산량이 적은 작업에 적합합니다. <br>
싱글 스레드이므로 컴퓨터의 자원을 적게 사용합니다. <br>
멀티 스레드에 비해 비교적 개발 난이도가 쉽습니다. <br>
자바스크립트로 작성하며 제이슨 형식과 쉽게 호환됩니다. <br>

<br>

* 단점 <br>
CPU 집약 작업에 적합하지 않습니다. <br>
싱글 스레드이므로 스레드가 죽으면 서버가 죽습니다. <br>

<br>

### 싱글 스레드 병렬
libuv 내부에 있는 이벤트 루프를 통해 비동기 방식으로 작업을 처리합니다. <br>

<br>

### 이벤트 기반 프로그래밍
특정 이벤트 발생에 의해 프로그램 흐름이 결정되는 패턴입니다. <br>
한 번에 하나의 코드를 순차적으로 실행하는 순차적 프로그래밍의 단점을 보완할 수 있습니다.

<br>

### 논 블로킹 I/O
이전 작업의 완료 여부와 관계없이 다음 코드가 실행됨을 의미합니다. <br>

<br>

### NPM YARN
자바스크립트 패키지 매니저를 의미합니다. <br>
패키지들의 설치와 제거 의존성 관계 등을 안전하게 수행할 수 있게 해줍니다. <br>

* NPM은 직렬 YARN은 병렬로 패키지를 설치합니다.
* NPM은 하나의 패키지가 의존하고 있는 다른 패키지들도 모두 설치하는 반면 YARN은 package.json 또는 yarn.lock 파일에 있는 패키지들만 설치합니다. <br>
* 둘의 성능 차이는 거의 없으며 취향에 따라 선택합니다.<br>

<br>

### dependency devDependency
프로덕션에서 사용할 패키지와 개발 시에 사용할 패키지를 의미합니다. <br>

* 런타임 단계에서 필요하면 dependency
* 컴파일 단계에서 필요하면 devDependency

<br>

### 익스프레스
노드의 http 내장 모듈의 단점을 보완한 웹 프레임워크입니다. <br>
라우터 미들웨어 등 서버 구성에 필요한 기능을 대부분 포함하고 있습니다. <br>

<br>

### 미들웨어
요청과 응답 중간에 위치하여 미들웨어라고 표현합니다. <br>
하나의 요청을 처리하기 위해 각자의 책임을 가지는 미들웨어를 조합해 사용하며 요청과 응답을 조작하여 기능을 추가하거나 필터링할 수 있습니다. <br>

<br>

### 라이브러리 프레임워크
1. 라이브러리 <br>
특정 기능을 모아둔 코드와 함수들의 집합을 의미합니다. <br>

2. 프레임워크 <br>
일정한 형태와 필요한 기능을 갖추고 있는 골격 뼈대를 의미합니다. <br>

<br>

* 차이점은 제어 흐름의 권한이 어디에 있느냐입니다. <br>
* 라이브러리는 사용자가 코드의 흐름을 직접 제어할 수 있습니다. <br>
* 프레임워크는 프레임워크의 코드 흐름을 통해 사용자의 코드가 활용됩니다. <br>

<br>

### ORM
객체와 데이터베이스의 릴레이션을 매핑해주는 도구입니다. <br>
데이터베이스의 접근을 프로그래밍 언어 관점에서 바라볼 수 있습니다. <br>

* 장점 <br>
데이터베이스 변경되었을 경우 변경된 문법을 ORM이 커버합니다. <br>
ORM은 독립적으로 작성되어 있으며 해당 객체들을 재활용할 수 있습니다. <br>
로우 쿼리의 절차적이고 순차적인 접근이 아닌 객체 지향 접근을 통해 생산성이 증가합니다. <br>

* 단점 <br>
프로젝트의 규모가 커질 시 난이도가 급격하게 올라갑니다. <br>
설계가 제대로 되지 않은 경우 성능 저하와 일관성이 무너질 수 있습니다. <br>

<br>

### SSR CSR
1. SSR <br>
서버에서 페이지를 구성하여 사용자에게 응답하는 방식입니다. <br>
서버에서 모든 데이터를 매핑하므로 CSR보다 페이지의 구성 속도는 늦지만 콘텐츠의 구성 완료 시점은 빠릅니다. <br>

<br>

2. CSR <br>
클라이언트에서 페이지를 구성하여 사용자에게 응답하는 방식입니다. <br>
SSR보다 초기 페이지 구성 속도는 빠르지만 서비스에 필요한 데이터를 추가로 요청해야 하므로 전체적인 페이지 완료 시점은 느립니다. <br>

<br>

### MVC
모델 뷰 컨트롤러로 이루어진 디자인 패턴입니다. <br>
애플리케이션 구성 요소를 3가지 역할로 나누어 개발 프로세스에서 각각의 구성 요소에만 집중하여 개발할 수 있다는 점과 재사용과 확장성이 용이하다는 장점이 있습니다. <br>

* 서비스 규모가 커질수록 모델과 뷰의 복잡성이 증가합니다. <br>

1. 모델 <br>
어떠한 동작을 수행할지 정의합니다. <br>
사용자가 편집을 원하는 모든 데이터를 가지고 있어야 합니다. <br>

<br>

2. 뷰 <br>
모델에게 질의해 결과값을 가져와 사용자에게 보여줍니다. <br>
모델이 가지고 있는 데이터를 저장해서는 안 됩니다. <br>

<br>

3. 컨트롤러 <br>
사용자 요청에 맞는 데이터를 모델에 요청해 결과를 View에 반영합니다. <br>

<br>

### 바디 쿼리 파람스
1. 바디 <br>
JSON XML Multi Form 형식의 데이터를 전송합니다. <br>

<br>

2. 쿼리 <br>
URL에 ? 이후에 데이터를 전송합니다. <br>
& 키워드를 이용해 여러 개의 데이터를 담을 수 있습니다. <br>

<br>

3. 파람스 <br>
주소에 포함된 데이터를 담습니다. <br>

<br>

* 리소스를 고유하게 식별해야 한다면 파람스 <br>
* 정렬이나 필터링을 해야 한다면 쿼리 <br>
* 전송 데이터양이 많고 직렬화된 이진 데이터일 경우 바디 <br>

<br>

### 객체 지향
객체들의 집합으로 프로그램의 상호 작용을 표현하고 데이터를 객체로 취급해 객체 내부의 메서드를 활용하는 소프트웨어 방법론입니다. <br>

1. 추상화 <br>
복잡한 시스템으로부터 핵심적인 개념 또는 기능을 추출해 내는 걸 의미합니다. <br>

<br>

2. 캡슐화 <br>
객체의 속성과 메서드를 하나로 매핑해 일부를 외부에 감추어 은닉하는 걸 의미합니다. <br>

* 은닉이란 다른 객체에게 자신의 정보를 숨기고 자신의 연산만을 통해 접근하는 방식을 의미합니다. 

<br>

3. 상속성 <br>
상위 클래스의 특성을 하위 클래스가 이어받아 재사용하거나 추가 또는 확장할 수 있는 방식을 의미합니다. 코드의 재사용과 유지 보수 측면에서 사용합니다. <br>

<br>

4. 다형성 <br>
하나의 클래스나 메서드가 상황에 따라 다양한 방법으로 동작하는 특징을 의미합니다. <br>
